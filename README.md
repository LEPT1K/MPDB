# MPDB — MITRE Parser Databases

**MPDB** (**MITRE_PARSER_DATABASES**) — комплексный инструмент для парсинга, обработки и анализа баз данных MITRE (CAPEC, CWE, CVE, ATT&CK) с удобным веб-интерфейсом.

## 📋 Возможности

### 🔹 Парсинг данных
- **CAPEC** - шаблоны атак
- **CWE** - типы уязвимостей
- **CVE** - известные уязвимости
- **MITRE ATT&CK** - тактики и техники противников

### 🔹 Обработка данных
- **Связывание** - автоматическое установление перекрестных ссылок между базами
- **Автозаполнение** - заполнение пустых полей: восстановление связей, извлечение фактов из описаний, detection из ATT&CK STIX, шаблонные фразы
- **Перевод** - быстрый многопоточный перевод выбранных полей на русский язык (Google Translate) с дедупликацией, упаковкой запросов и кэшированием
- **Нормализация** - очистка и стандартизация данных

### 🔹 Графический интерфейс
- **Панель управления** - запуск и мониторинг всех процессов в реальном времени, быстрый доступ к PDF-отчёту и локальной папке `output/`
- **Редактор баз данных** - просмотр, поиск и редактирование записей
- **Связи** - интерактивный граф перекрёстных связей записи (глубина 1-3 шага) с детальной карточкой записи по клику и переходом в редактор, а также поиск пути между двумя записями через все 4 базы
- **Редактор кэша переводов** - пагинация, поиск, сворачивание длинных текстов
- **Настройки** - конфигурация лимитов парсинга и параметров перевода

## 🚀 Быстрый старт

### 1. Установка зависимостей

```bash
pip install -r requirements.txt
```

### 2. Настройка (опционально)

Отредактируйте `src/config.py` для настройки:
- Источников данных
- Параметров перевода (потоки, задержка)
- Шаблонов автозаполнения (`AUTOFILL_TEMPLATES`)
- Ограничений на количество записей

### 3. Запуск

#### Вариант А: Графический интерфейс (рекомендуется)

```bash
python gui/run_gui.py
```

Затем откройте браузер: **http://localhost:5000**

#### Вариант Б: Консольное меню

```bash
python src/menu.py
```

#### Вариант В: Прямой запуск всех этапов

```bash
python src/run_all.py
```

## 🎯 Подробное описание

### Этапы обработки

1. **Парсинг (Parsing)**
   - Скачивание актуальных данных из официальных источников MITRE
   - Разбор XML/JSON форматов
   - Извлечение ключевых полей
   - Сохранение в JSON

2. **Связывание (Cross-Linking)**
   - Анализ перекрестных ссылок между базами
   - Заполнение полей related_capec, related_cwe, related_mitre
   - CAPEC → ATT&CK извлекается напрямую из `Taxonomy_Mappings` (Taxonomy_Name="ATTACK") при парсинге, остальные связи дозаполняются на этом этапе
   - Добавление мер защиты (mitigations)

3. **Перевод (Translation)**
   - Многопоточный перевод выбранных полей на русский язык через Google Translate
   - Глобальная дедупликация строк и упаковка коротких строк в один запрос — меньше запросов к Google и кратно быстрее первый прогон
   - Постоянный кэш — повторные запуски почти мгновенны

4. **Автозаполнение (Autofill)** — выполняется **после перевода** (работает по русским словарям/шаблонам), ярусная стратегия от точного к шаблонному:
   - **Ярус A**: восстановление связей по реальным ID (реверс и транзитивность, например ATT&CK → CAPEC → CWE)
   - **Ярус B**: извлечение фактов из описаний — порты, сервисы, тип атаки, категория CWE, severity из CVSS
   - **Ярус C**: detection для ATT&CK из STIX-первоисточника (detection strategies → analytics) с переводом
   - **Ярус D**: шаблонные фразы (настраиваются в `Config.AUTOFILL_TEMPLATES`) для оставшихся пустых текстовых полей
   - Поля-связи (related_*) шаблонами не заполняются — только реальные ID

### Графический интерфейс

Веб-интерфейс предоставляет:

- **Дэшборд** с карточками статистики, лентой обновлений MITRE, управлением процессами, скачиванием PDF-отчёта и кнопкой быстрого перехода в локальную папку `output/` с JSON-базами
- **Редактор баз данных** с поиском, фильтрацией и пагинацией
- **Страницу "Связи"** с графом перекрёстных связей записи (с настраиваемой глубиной 1-3 шага и детальной карточкой записи прямо под графом) и поиском пути между двумя записями через CAPEC/CWE/CVE/ATT&CK — подробнее см. [docs/features/link-graph.md](docs/features/link-graph.md)
- **Редактор кэша переводов** с пагинацией, поиском и сворачиванием длинных текстов
- **Страницу настроек** для конфигурации лимитов и перевода
- **Real-time логи** с цветовым кодированием событий

### Консольное меню

Альтернативный интерфейс для управления процессами:

```
1. Скачать и распарсить базы (без перевода)
2. Связать данные (заполнить перекрёстные ссылки)
3. Перевести выбранные поля (Google Translate)
4. Автозаполнение пустых полей (связи/факты/STIX/шаблоны)
5. Запустить все этапы последовательно
6. Очистить кэш перевода
7. Очистить некачественные переводы из кэша
8. Настроить параметры перевода
0. Выход
```

## ⚙️ Конфигурация

Основные настройки в `src/config.py`:

```python
# Перевод
ENABLE_TRANSLATION = False          # Включить перевод при парсинге
TRANSLATE_TO = "ru"                 # Целевой язык
TRANSLATION_DELAY = 0.2             # Базовая задержка перед запросом (применяется джиттер ±50%)
TRANSLATION_WORKERS = 8             # Число параллельных потоков перевода
TRANSLATION_MAX_RETRIES = 5         # Число повторных попыток при ошибке
TRANSLATION_SERVICE = "google"      # перевод выполняется через Google Translate

# Автозаполнение — шаблонные фразы для пустых полей
AUTOFILL_TEMPLATES = { ... }        # см. config.py

# Ограничения (0 = все записи)
MAX_CAPEC_RECORDS = 1000
MAX_CWE_RECORDS = 1000
MAX_CVE_RECORDS = 4000
MAX_ATTACK_RECORDS = 1000
```

## 📊 Выходные данные

После обработки в папке `output/` создаются:

- `capec_database.json` - база шаблонов атак
- `cwe_database.json` - база типов уязвимостей
- `cve_database.json` - база известных уязвимостей
- `mitre_attack.json` - база тактик и техник ATT&CK
- `translate_cache.json` - кэш переводов

Каждая запись содержит:
- ID
- Название
- Описание
- Связанные записи из других баз
- Меры защиты
- И другие поля в зависимости от типа базы

## 📑 Описание полей JSON-файлов

Ниже приведено полное описание полей каждого JSON-файла из папки `output/`. Поля-связи (`related_*`) изначально заполняются парсерами из первоисточников, затем дополняются на этапе **Связывания** (`cross_linker.py`) и **Автозаполнения** (`step4_autofill.py`).

### `capec_database.json` — шаблоны атак (CAPEC)

Список записей, каждая описывает один шаблон атаки.

| Поле | Тип | Описание |
|---|---|---|
| `id` | строка | Идентификатор шаблона атаки, формат `CAPEC-<номер>` (например, `CAPEC-1`) |
| `name` | строка | Название шаблона атаки |
| `description` | строка | Подробное описание шаблона атаки |
| `severity` | строка | Типичная серьёзность атаки: `LOW`, `MEDIUM`, `HIGH`, `VERY_HIGH` или `UNKNOWN`, если не указана в первоисточнике |
| `related_cwe` | список строк | Связанные уязвимости из CWE (`CWE-<номер>`), которые может использовать данный шаблон атаки |
| `related_mitre` | список строк | Связанные техники MITRE ATT&CK (`T<номер>` или `T<номер>.<подномер>`). Изначально берётся из `Taxonomy_Mappings` (ATTACK) в первоисточнике CAPEC, далее дополняется при связывании на основе `related_capec` в ATT&CK |
| `prerequisites` | список строк | Предварительные условия, необходимые для успешного выполнения атаки. Если в первоисточнике не указаны — заполняется шаблонной фразой на этапе автозаполнения (ярус D) |
| `mitigations` | список строк | Меры противодействия (защиты) от данной атаки. Если в первоисточнике не указаны — заполняется шаблонной фразой (ярус D) |

### `cwe_database.json` — типы уязвимостей (CWE)

Список записей, каждая описывает один тип слабости/уязвимости программного обеспечения.

| Поле | Тип | Описание |
|---|---|---|
| `id` | строка | Идентификатор уязвимости, формат `CWE-<номер>` (например, `CWE-1004`) |
| `name` | строка | Название типа уязвимости |
| `description` | строка | Описание уязвимости |
| `category` | строка | Категория/класс уязвимости из первоисточника (атрибут `Class` или `Type`, в нижнем регистре), например `base`, `class`, `variant` |
| `related_capec` | список строк | Связанные шаблоны атак из CAPEC (`CAPEC-<номер>`), которые используют данную уязвимость |
| `mitigation` | строка | Меры противодействия. В отличие от других баз — это **одна строка**, а не список: объединяет до 3 первых мер из первоисточника (через `; `), а при связывании дополняется мерами из связанных CAPEC/ATT&CK (через перевод строки). Если изначально пусто — заполняется шаблонной фразой (ярус D) |
| `requires_technology` | список строк | Платформы/технологии, к которым применима уязвимость (из `Applicable_Platforms`, в нижнем регистре). Если список пуст — заполняется шаблонной фразой "не зависит от конкретной технологии" (ярус D) |
| `detection_methods` | список строк | Методы обнаружения уязвимости (нормализованные идентификаторы, например `automated_static_analysis`). Если список пуст — заполняется шаблонной фразой (ярус D) |

### `cve_database.json` — известные уязвимости (CVE)

Список записей, каждая описывает одну зарегистрированную уязвимость (NVD).

| Поле | Тип | Описание |
|---|---|---|
| `id` | строка | Идентификатор CVE, формат `CVE-<год>-<номер>` (например, `CVE-2012-6435`) |
| `description` | строка | Описание уязвимости (английское описание из NVD; русский текст появляется на этапе перевода) |
| `severity` | строка | Уровень критичности по CVSS: `LOW`, `MEDIUM`, `HIGH`, `CRITICAL` или `UNKNOWN` |
| `cvss_score` | число | Базовая оценка CVSS (0.0–10.0). Берётся из метрик `cvssMetricV31` → `cvssMetricV30` → `cvssMetricV2` (первая найденная) |
| `affected_software` | список строк | Затронутое программное обеспечение/прошивки (производитель и продукт), извлечённые из CPE-конфигураций (максимум 5). Если список пуст — заполняется фразой "не указано в первоисточнике" (ярус D) |
| `attack_type` | строка | Тип атаки, определённый эвристически по описанию: `sql_injection`, `cross_site_scripting`, `remote_code_execution`, `brute_force`, `denial_of_service`, `privilege_escalation`, `deserialization_attack` или `unknown` |
| `related_cwe` | список строк | Связанные типы уязвимостей CWE (`CWE-<номер>`), указанные в NVD для данной CVE |
| `related_capec` | список строк | Связанные шаблоны атак CAPEC (`CAPEC-<номер>`). Изначально пусто, заполняется при связывании по цепочке CVE → CWE → CAPEC |
| `related_mitre` | список строк | Связанные техники MITRE ATT&CK (`T<номер>`). Изначально пусто, заполняется при связывании по цепочке CVE → CWE → CAPEC → ATT&CK |
| `mitigations` | список строк | Меры противодействия (до 10), собранные из мер защиты связанных CWE/CAPEC/ATT&CK. Если пусто — заполняется шаблонной фразой "рекомендуется обновить уязвимое ПО до исправленной версии" (ярус D) |
| `requires_service` | список строк | Сервисы, требуемые для эксплуатации уязвимости, определённые эвристически по описанию (например `web_server`, `database`, `ssh`, `rdp`, `smb`, `ftp`, `exchange_server`, `jenkins`). Если пусто — заполняется фразой "не определено" (ярус D) |
| `requires_port` | список чисел | Номера сетевых портов (1–65535), упомянутые в описании уязвимости (например, `2222`, `44818`). Если пусто — заполняется фразой "не определено" (ярус D) |
| `prerequisites` | список строк | Предварительные условия для эксплуатации. Изначально пусто; может заполняться на этапе автозаполнения (ярус B/D) |

### `mitre_attack.json` — тактики и техники ATT&CK

Список записей, каждая описывает одну технику (или подтехнику) MITRE ATT&CK.

| Поле | Тип | Описание |
|---|---|---|
| `id` | строка | Идентификатор техники ATT&CK, формат `T<номер>` или `T<номер>.<подномер>` для подтехник (например, `T1055.011`) |
| `name` | строка | Название техники |
| `tactic` | строка | Тактика (этап kill chain), к которой относится техника, например "Повышение привилегий". Если у подтехники не указана — наследуется от родительской техники |
| `description` | строка | Описание техники (HTML-теги удалены). Если у подтехники пусто — наследуется от родительской техники |
| `platforms` | список строк | Платформы, на которые распространяется техника (например `Windows`, `Linux`, `macOS`). Если у подтехники не указаны — наследуются от родительской техники |
| `related_cwe` | список строк | Связанные типы уязвимостей CWE (`CWE-<номер>`), указанные во внешних ссылках техники |
| `related_capec` | список строк | Связанные шаблоны атак CAPEC (`CAPEC-<номер>`) — из внешних ссылок техники, а также дополняется при связывании на основе `related_mitre` из CAPEC |
| `requires_service` | список строк | Сервисы/технологии, необходимые для применения техники, определённые эвристически по платформам и описанию (например `windows_os`, `api_service`, `active_directory`, `powershell`). Если пусто — заполняется фразой "не определено" (ярус D) |
| `detection` | строка | Рекомендации по обнаружению использования техники (поле `x_mitre_detection` из STIX, HTML очищен). Если у подтехники пусто — наследуется от родителя, либо дополняется на этапе автозаполнения (ярус C) из detection-стратегий ATT&CK. Если всё равно пусто — заполняется шаблонной фразой (ярус D) |
| `mitigations` | список строк | Меры противодействия — тексты объектов `course-of-action` из STIX, связанных с техникой отношением `mitigates` (у подтехник без своих мер наследуются меры родителя). При связывании дополняется мерами связанных CAPEC. Если пусто — заполняется шаблонной фразой (ярус D) |

### `translate_cache.json` — кэш переводов

Объект-словарь (key-value), используемый модулем `translator.py` для кэширования переводов и избежания повторных запросов к Google/Yandex Translate.

| Элемент | Описание |
|---|---|
| Ключ | Оригинальный текст (как правило, на английском языке) |
| Значение | Переведённый текст (на русский язык) |

### `updates_cache.json` — кэш ленты обновлений MITRE баз

Объект с единственным полем `updates` — списком записей о состоянии источников данных (используется на дэшборде GUI).

| Поле | Тип | Описание |
|---|---|---|
| `id` | строка | Уникальный идентификатор записи (источник + временная метка) |
| `source` | строка | Короткий код источника: `capec`, `cwe`, `attack`, `cve` |
| `name` | строка | Отображаемое название источника (например, `CAPEC`, `CWE`, `MITRE ATT&CK`, `NVD CVE`) |
| `description` | строка | Краткое описание источника данных |
| `url` | строка | Ссылка на официальный источник данных |

## 🆘 Решение проблем

### GUI не запускается
```bash
pip install -r gui/requirements.txt
```

### Ошибки при парсинге
- Проверьте подключение к интернету
- Увеличьте таймауты в `config.py`
- Попробуйте запустить с меньшим количеством записей

### Проблемы с переводом
- Проверьте API ключ для Yandex (если используется)
- При ошибках 429 (Too Many Requests) уменьшите `TRANSLATION_WORKERS` или увеличьте `TRANSLATION_DELAY`

### Автозаполнение не скачивает STIX
- Ярус C требует доступ к GitHub (~50 МБ). Можно указать локальный файл: `python src/step4_autofill.py --stix путь/к/enterprise-attack.json` или пропустить ярус: `--no-detection`

## 🔒 Безопасность при развёртывании

GUI предназначен для **локального использования** (на `localhost`). Перед тем как делать его доступным из сети — учтите следующее:

- **Сервер слушает все интерфейсы**: `socketio.run(app, host='0.0.0.0', port=5000, ...)` в `gui/run_gui.py` и `gui/app.py`. Это значит, что приложение доступно любому устройству в той же сети (а при проброшенном порте — и из интернета), а не только с этого компьютера.
- **Это dev-сервер Werkzeug** (`allow_unsafe_werkzeug=True`) — без защиты от нагрузки, лимитов запросов и прочих механизмов, обязательных для production.
- **Нет аутентификации**: любой, кто достучится до порта, получает полный доступ ко всем функциям без логина и пароля:
  - запуск/остановку процессов парсинга, связывания, перевода (нагрузка на CPU и сеть вашего компьютера, обращения к MITRE/NVD от его имени);
  - `PUT /api/database/<db_name>` и `PUT /api/database/<db_name>/record/<record_id>` — запись произвольного JSON в файлы `output/`. Имя базы (`db_name`) подставляется в путь к файлу без проверки по белому списку;
  - `POST /api/output-folder/open` — запускает процесс (проводник/файловый менеджер) на хост-машине по сетевому запросу.

### Рекомендации
- Для локальной работы используйте `http://localhost:5000` и не открывайте порт 5000 в роутере/файрволе наружу.
- Если нужен доступ из локальной сети (например, для лабораторной работы по безопасности), запускайте только в изолированном сегменте/VM и добровольно ограничивайте доступ (firewall-правило, разрешающее только нужные IP).
- Для доступа за пределами `localhost` добавьте базовую аутентификацию или поставьте перед приложением reverse proxy (nginx, Caddy) с авторизацией.
- Не выставляйте этот GUI в открытый интернет — он специально не предназначен для этого и является хорошей учебной целью именно как пример незащищённого dev-сервера.
