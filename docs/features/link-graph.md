# 🔗 Граф связей записи и поиск пути между записями

Интерактивная визуализация перекрёстных ссылок между CAPEC, CWE, CVE и MITRE ATT&CK. Доступна на странице **"Связи"**.

## Возможности

### Граф связей записи

- Поиск записи по id или названию (например, `CAPEC-66`, `CWE-89`, `CVE-2021-44228`, `T1059`)
- Переключатель глубины обхода: **1, 2 или 3 шага** от выбранной записи (BFS по перекрёстным ссылкам)
- Цветовая кодировка узлов по базе — единая на страницах "Главная" и "Связи":
  - CAPEC — синий (`#0d6efd`)
  - CWE — жёлтый (`#ffc107`)
  - CVE — зелёный (`#198754`)
  - ATT&CK — красный (`#dc3545`)
- **Детали записи прямо под графом**: клик средней кнопкой мыши (колесо) по узлу открывает карточку "Граф связей записи" с кратким описанием
- Кнопка **"В базе данных"** в карточке деталей — переход в редактор соответствующей базы данных с открытой записью

### Путь между записями

- Поиск кратчайшей цепочки связей между двумя записями (например, между CVE и техникой ATT&CK)
- Поля **"Откуда"** и **"Куда"** с автодополнением по id/названию из всех 4 баз
- Результат — цепочка карточек с указанием базы каждого узла на пути
- Поиск ограничен 6 шагами; для части записей (особенно техник ATT&CK и некоторых CAPEC/CWE без перекрёстных связей в данных MITRE) путь может не существовать — это явно отображается
- В UI есть сворачиваемая подсказка с объяснением назначения функции и кратким гайдом по использованию

## Зачем это нужно

Базы CAPEC, CWE, CVE и ATT&CK редко связаны напрямую — обычно между ними цепочка из нескольких шагов (например, `CVE → CWE → CAPEC → ATT&CK`). Граф и поиск пути визуализируют эти цепочки автоматически, что удобно для анализа угроз: по уязвимости понять, какими шаблонами атак и техниками противника она может быть использована, и наоборот.

## Архитектура

| Компонент | Файл | Назначение |
|---|---|---|
| Backend | `src/link_graph.py` | Класс `LinkGraph` — индекс перекрёстных связей, `get_network()`, `find_path()`, `search_nodes()`, поиск по id/названию (`_resolve_id`), кэш с обновлением по mtime файлов БД |
| API | `gui/app.py` | `/api/linking/*`, `/api/database/<db_name>/record/<record_id>` |
| UI | `gui/templates/index.html` | Страница "Связи": граф, карточка деталей записи, секция "Путь между записями" |
| JS | `gui/static/js/app.js` | `renderLinkGraphNetwork`, `showLinkGraphNodeDetails`, `renderLinkGraphNodeDetails`, `openRecordInDatabase`, поиск пути и рендер цепочки |
| CSS | `gui/static/css/style.css` | Стили графа (`.link-graph-network`), карточки деталей (`#linkGraphNodeDetails`), цепочки пути (`.path-chain*`) |

## API

```
GET /api/linking/stats              # статистика перекрёстных связей между базами
GET /api/linking/search?q=...       # поиск записей по id/названию для графа
GET /api/linking/graph/<node_id>?depth=1..3   # локальный граф связей узла
GET /api/linking/path?from=...&to=...         # кратчайший путь между двумя записями
GET /api/database/<db_name>/record/<record_id> # полная запись БД (для карточки деталей)
```

`node_id` принимает id записи (например, `CAPEC-66`, `CWE-89`, `CVE-2021-44228`, `T1059`) — точное соответствие базе определяется по формату идентификатора.

## CAPEC ↔ ATT&CK

Связи CAPEC → ATT&CK теперь извлекаются напрямую из `Taxonomy_Mappings` (`Taxonomy_Name="ATTACK"`, `Entry_ID`) на этапе парсинга CAPEC (`src/parsers/capec_parser.py`), что даёт точные прямые связи `related_mitre` для большинства записей CAPEC. Остальные перекрёстные связи (CAPEC↔CWE, CWE↔CVE и реверсивные/транзитивные) дозаполняются на этапах "Связывание" и "Автозаполнение" (ярус A).
